# SQLVagrantLab

A turnkey lab environment that provisions **Windows Server virtual machines pre-installed with SQL Server Developer Edition** using [Packer](https://www.packer.io/) and [Vagrant](https://www.vagrantup.com/) on QEMU/KVM.

The lab topology is defined in a single [`config.yaml`](config.yaml) file. Running one command builds the full environment: Active Directory domain controller + SQL Server node, joined to the domain with SQL Server configured and ready for use.

---

## Prerequisites

Before deploying, ensure the following are installed and configured on the host:

| Requirement | Notes |
|---|---|
| QEMU + KVM | `sudo apt install qemu-system-x86`; user must be in the `kvm` group |
| Vagrant | `sudo apt install vagrant` or install from [vagrantup.com](https://vagrantup.com) |
| sshpass | `sudo apt install sshpass` (used for headless VM authentication) |
| vagrant-qemu plugin | `vagrant plugin install vagrant-qemu` |
| vagrant-reload plugin | `vagrant plugin install vagrant-reload` |
| PowerShell 7 (`pwsh`) | Required to run `Deploy-Lab.ps1` |
| Pre-built `.box` files | Built by Packer — see [`packer/README.md`](packer/README.md) |

> **Important**: The `Install-Prerequisites.ps1` script will automate the installation of `qemu`, `vagrant`, `pwsh`, and the required Vagrant plugins on a fresh Linux machine. 
> 
> ```bash
> sudo pwsh -File Install-Prerequisites.ps1
> ```
> *Note: If you do not run this, you must manually install the `vagrant-qemu` and `vagrant-reload` plugins before continuing.*

## Deploying the Lab

Run the master deployment script from the project root:

```bash
sudo pwsh -File Deploy-Lab.ps1
```

This script:
1. Reads [`config.yaml`](config.yaml) to get the lab topology
2. Validates that all required Packer `.box` files exist in `/opt/vagrant-boxes/`
3. Generates a `Vagrantfile` from the ERB template in `vagrant/`
4. Runs `vagrant up --provider=qemu --no-parallel`

> **Note:** `sudo` is required for QEMU/KVM access. All Vagrant operations are run as root.

### Additional options

```bash
# Preview the generated Vagrantfile without starting VMs
sudo pwsh -File Deploy-Lab.ps1 -WhatIf

# Use a different topology file
sudo pwsh -File Deploy-Lab.ps1 -ConfigPath ./my-config.yaml
```

## Configuring the Lab Topology

Edit [`config.yaml`](config.yaml) to change the lab topology before running `Deploy-Lab.ps1`:

```yaml
global:
  domain_name: test.dev
  admin_user: admin
  admin_password: "P@ssw0rd"
  box_library_path: "/opt/vagrant-boxes"

regions:
  - name: RegionA
    network: 192.168.10.0/24
    nodes:
      - hostname: dc01
        role: DomainController
        os_version: "2022"
        sql_version: "2022"
        static_ip: 192.168.10.10
        cpus: 2
        memory_mb: 2048
      - hostname: sql01
        role: SQLServer
        os_version: "2022"
        sql_version: "2022"
        static_ip: 192.168.10.20
        cpus: 4
        memory_mb: 4096
```

Re-run `Deploy-Lab.ps1` after any changes to regenerate the `Vagrantfile`.

## Common Operations

```bash
# SSH into a running VM
vagrant ssh dc01

# Shut down all VMs (preserves disk state)
sudo vagrant halt

# Destroy all VMs and disks (full teardown)
sudo vagrant destroy -f

# Re-provision a running VM
sudo vagrant provision dc01
```

## Troubleshooting

### WinRM connection hangs on first boot

On first boot after Sysprep, the `LocalAccountTokenFilterPolicy` registry key is cleared by Windows. Without it, WinRM rejects remote connections from the `vagrant` user even with correct credentials.

If `vagrant up` is stuck at:
```
dc01: Warning: Authentication failure. Retrying...
```

**Do not stop `vagrant up`.** Open a second terminal and SSH into the VM:
```bash
ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null -p 50022 vagrant@localhost
```
*Password: `vagrant`*

Then run the following PowerShell commands inside the VM:
```powershell
New-ItemProperty `
  -Path 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System' `
  -Name LocalAccountTokenFilterPolicy `
  -Value 1 -PropertyType DWORD -Force

Restart-Service WinRM
```

Leave the SSH session open and return to the original terminal — `vagrant up` should immediately continue connecting.

> This workaround is only needed for the current box build. The next Packer rebuild will include a startup scheduled task that applies this setting automatically on first boot.

---

### Port conflict on startup

If Vagrant reports that port `50022` (or `50023`) is already in use, a zombie QEMU process from a previous run may be holding it:

```bash
sudo lsof -i :50022
sudo kill -9 <pid>
```

---

### VM crashes with INACCESSIBLE_BOOT_DEVICE (BSOD)

The QEMU machine type and storage controller in the Vagrantfile **must exactly match** the Packer build settings. If you change hardware settings (e.g. `qe.machine`, `qe.drive_interface`), you must fully destroy and re-import the VM:

```bash
sudo vagrant destroy -f
sudo vagrant up --provider=qemu --no-parallel
```

---

### Visual debugging (QEMU GUI)

If the VM won't boot and you need to see the screen, enable the QEMU display by temporarily modifying `qe.extra_qemu_args` in the `Vagrantfile`:

```ruby
qe.extra_qemu_args = ["-enable-kvm", "-display", "gtk", "-vga", "std", ...]
```

Then re-run `vagrant up`. A desktop window will appear showing the boot sequence.
